# Multi-env setup stub

Generated by `firebase-multi-env init`.

**Production path:** pinned Functions + one service account per env + per-env secrets + deploy isolation.
Logical (unpinned) mode is for local/emulators only.

## 1. Copy rules snippets

- Gated Firestore DBs: `firestore.rules.snippets/firestore.gated.rules.snippet`
- Public/prod Firestore: `firestore.rules.snippets/firestore.public.rules.snippet`
- Gated Storage bucket: `firestore.rules.snippets/storage.gated.rules.snippet`
- Public Storage bucket: `firestore.rules.snippets/storage.public.rules.snippet`

Wire each database / bucket to its own rules file in `firebase.json`. Prefer a **bucket per env** — do not share prod Storage with gated envs.

## 2. Follow project parity

Read in order:

1. `multi-env/PROJECT_PARITY.md` — checklist + CI gate
2. `multi-env/ISOLATION.md` — pinned vs logical
3. `multi-env/PROVISION.md` — generate per-env SA / bucket / secret scripts
4. `multi-env/iam-sa-per-env.md` — runtime SAs
5. `multi-env/secrets-per-env.md` — Secret Manager
6. `multi-env/deploy-isolation.md` — CI / WIF / Hosting

Also see `functions.pinned.qual.example.ts`, `firebase.codebases.example.json`, and `github-actions.deploy.example.yml`.

Generate IAM scripts (does not call GCP until you run them):

```bash
npx firebase-multi-env provision --project YOUR_PROJECT --envs production,qual
bash multi-env/provision/provision.all.sh
```

Auth stays **shared**; gate non-prod with `grant-env` claims.

| Mode | When | Config |
|---|---|---|
| **Pinned** (production) | One deploy + SA per env | `pinned: true`, `APP_ENV=<env>` |
| **Logical** (local only) | One process serves many envs | `pinned: false` (blocked on Cloud deploy by default) |

## 3. Configure Cloud Functions (pinned)

```ts
import { createEnvRuntime, createGetDb, createGetDbForEnv, requireAuth } from 'firebase-multi-env/server';
import { createWithAppEnvV2 } from 'firebase-multi-env/functions-v2';

export const appEnvRuntime = createEnvRuntime({
  pinned: true,
  pinnedEnvironment: process.env.APP_ENV, // "qual" for the qual deploy
  environments: {
    production: {
      database: '(default)',
      origins: ['https://YOUR_PROD_HOST.web.app'],
    },
    qual: {
      database: 'qual-env',
      origins: ['https://YOUR_QUAL_HOST.web.app'],
      requireClaim: true,
    },
  },
});

export const getDb = createGetDb(appEnvRuntime);
export const getDbForEnv = createGetDbForEnv(appEnvRuntime);
export const withAppEnv = createWithAppEnvV2(appEnvRuntime);
```

Attach a per-env `serviceAccount` on each `onCall` / `onRequest` (see `functions.pinned.qual.example.ts`).

For raw HTTP handlers, verify ID tokens:

```ts
import { createWithAppEnvHttp } from 'firebase-multi-env/http';

const withHttp = createWithAppEnvHttp(appEnvRuntime, { verifyIdToken: true });
```

### Local / emulator only (unpinned)

```ts
export const appEnvRuntime = createEnvRuntime({
  pinned: false,
  environments: { /* same map */ },
});
```

Deployed Cloud Functions throw if unpinned unless you set `allowUnpinnedCloudDeploy: true`.

## 4. Configure the web client

```ts
import { createMultiEnvClient } from 'firebase-multi-env/client';

const appEnv = import.meta.env.VITE_APP_ENV;
const { callable, getDb } = createMultiEnvClient({
  app,
  functions: getFunctions(app),
  appEnv,
  databases: {
    production: '(default)',
    qual: 'qual-env',
  },
});
```

## 5. Grant gated access

```bash
npx firebase-multi-env grant-env qual --project YOUR_PROJECT you@email.com
```

## 6. Sanity-check (CI gate)

```bash
npx firebase-multi-env doctor --strict
```

import multiEnv from 'firebase-multi-env/eslint';
